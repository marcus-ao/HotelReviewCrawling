# 基于酒店评论知识库与高效语言模型微调的推荐智能体构建研究

## ——毕业设计完整设计与实施路线规划说明文档

---

## 1. 课题一句话定义

本课题拟构建一个**面向酒店推荐场景的对话式推荐系统原型**：以酒店评论为数据基础，先抽取“方面—情感—证据”形成细粒度知识库，再通过结构化筛选与 RAG 检索提供可验证证据，最后由经过 PEFT/QLoRA 轻量微调的小语言模型完成偏好理解、澄清交互和推荐解释生成。由于该系统的流程基本固定、状态需要持续维护，因此在工程实现上更适合被定义为**有状态推荐工作流**，而不是开放式自主 agent；LangGraph 官方也明确将其定位为适合构建长生命周期、可持久化、可调试的有状态 workflow/agent 基础设施，而 LangGraph 文档同时区分了“workflow 的预定义流程”与“agent 的动态决策流程”。([LangChain Docs][1])

---

## 2. 课题核心目标

### 2.1 总体目标

在有限算力条件下，设计并实现一个能够完成以下任务的酒店推荐系统原型：

1. 理解用户自然语言需求；
2. 在酒店评论中检索与需求相关的证据；
3. 依据证据生成可解释推荐；
4. 在多轮对话中通过澄清与反馈逐步修正推荐结果；
5. 通过对比实验验证两条研究问题。

### 2.2 两个核心研究问题

#### RQ1：细粒度方面知识库是否比原始评论直接 RAG 更能提升推荐解释的可信度与可验证性？

这个问题关注的重点不是“推荐文案是否更流畅”，而是**知识组织方式**是否会影响证据检索质量、解释 groundedness 和用户对推荐可信度的感知。RAGAs 明确指出，RAG 系统评估应同时考虑检索上下文质量、回答忠实度以及最终输出质量；LangSmith 的 RAG 评测文档也将 retrieval quality、answer relevance、answer accuracy 等维度分开看待。([ACL Anthology][2])

#### RQ2：PEFT 微调对会话式推荐的主要收益，是否体现在偏好解析和澄清交互，而非事实性知识？

这个问题要验证的是：当酒店事实主要由知识库和检索模块提供时，PEFT 的主要价值是否体现在**用户需求结构化、澄清提问、约束遵守、证据式表达**等行为能力，而不是“把酒店知识记进模型参数”。PEFT 文档将 LoRA/QLoRA 定位为一种低成本任务适配手段；其量化文档也明确说明，量化与 PEFT 结合是单卡训练大模型的重要方式，QLoRA 即“将模型量化到 4-bit，再在其上训练 LoRA 适配器”。([Hugging Face][3])

## 3. 课题总体思路：把系统“拆开做”

你现在最需要建立的不是“多堆几个热门名词”，而是一个非常清晰的系统分层观：

### 3.1 三层系统结构

#### 第一层：事实层（知识库）

负责从评论中沉淀出**可检索、可统计、可验证**的事实依据，包括：

* 酒店基础信息；
* 评论原文与句子切分结果；
* 方面类别；
* 方面情感极性；
* 代表性证据句；
* 向量索引。

#### 第二层：决策层（筛选与检索）

负责把用户自然语言需求转换成可执行约束，并进行：

* 结构化过滤；
* 候选酒店排序；
* 证据句召回与重排。

#### 第三层：交互层（LLM）

负责：

* 偏好解析；
* 澄清交互；
* 证据约束下的推荐生成；
* 多轮反馈后的约束更新。

这个划分符合当前 LLM+RAG 系统“检索与生成分开设计、分开评估”的常见实践，也更适合毕业设计在有限时间内分模块推进。([ACL Anthology][2])

## 4. 本课题的最终系统定位

### 4.1 系统不是“纯聊天机器人”

它的本质是一个**任务型、推荐型、证据驱动的对话系统**。

### 4.2 系统也不是“纯推荐模型”

它不是只给一个酒店列表，而是要做到：

* 能理解用户约束；
* 能说明推荐理由；
* 能解释证据来源；
* 能根据后续反馈继续调整推荐。

### 4.3 最准确的工程定位

建议你在论文里将其表述为：

> **基于细粒度评论知识库、PEFT 微调语言模型与 RAG 机制的有状态会话式酒店推荐工作流**

这样既保留“推荐智能体”的外部说法，又避免“agent”定义过宽带来的实现压力。LangGraph 官方关于 workflow 与 agent 的区分，正好可以支持你采用更稳妥的 workflow 架构。([LangChain Docs][1])

## 5. 技术路线总览

```text

   ↓

   ↓
PEFT 微调小模型生成推荐解释
   ↓
用户反馈
   ↓
约束更新与再次推荐
```

## 6. 核心设计原则

### 6.1 知识外置，能力内化

这是整个课题最关键的设计原则。

* **知识库 / RAG 负责事实**
* **微调模型负责行为能力**

也就是说：

* 酒店是否安静、卫生如何、地理位置怎么样，这些信息尽量放在外部知识库里；
* 模型负责“怎么理解用户需求、怎么问问题、怎么组织证据、怎么生成解释”。

这样做的原因是：评论数据动态、噪声大、更新快，外置知识更便于更新与验证；而 PEFT 天然更适合做行为适配。([Hugging Face][3])

### 6.2 先做“可控系统”，再做“更智能系统”

毕业设计最怕一开始追求“全自动、全开放、全智能”，结果系统不可控。
因此建议你优先做：

* 封闭方面体系；
* 明确约束解析；
* 有控制的证据生成；
* 固定流程的多轮交互。

### 6.3 实验优先于炫技

本课题最终能否成立，不在于你用了多少框架，而在于是否能通过实验回答那两个研究问题。

## 7. 各模块完整设计

---

## 7.1 模块一：细粒度方面知识库构建

### 7.1.1 模块目标

从原始评论中构建一个兼具**结构化统计能力**与**语义检索能力**的酒店评论知识底座。

### 7.1.2 数据输入

建议数据来源分两类：

#### A. 主数据源

优先使用公开酒店评论数据集，保证合法合规、结构稳定、便于复现。

#### B. 补充数据源

若条件允许，在合规前提下补充中文 OTA 场景数据，但不建议把爬虫部分作为论文重点。

### 7.1.3 预处理流程

1. 去 HTML、表情、广告噪声；
2. 删除过短或信息量极低的评论；
3. 按句号、分号、感叹号等进行中文分句；
4. 统一酒店主键；
5. 规范价格、区域、星级等字段。

### 7.1.4 方面抽取策略

PyABSA 是一个面向 ABSA 任务的开源框架，支持 APC、ATEPC、ASTE 等任务，并提供训练、推理、增强和可视化能力。对本课题而言，它适合承担方面级情感分析的实验入口。([pyabsa.readthedocs.io][4])

但从毕业设计风险控制角度，**不建议一开始就追求完全开放式三元组抽取**。
更推荐的做法是：

### 7.1.5 建议采用“封闭方面体系”

先定义 6–8 个固定方面：

* 位置交通
* 卫生环境
* 服务质量
* 设施舒适度
* 性价比
* 安静程度
* 亲子友好
* 商务适配

然后执行两步：

1. 评论句映射到一个或多个方面；
2. 对每个方面判定极性（正向 / 负向 / 中性）。

这样比直接做开放式 ASTE 更稳，更容易人工校验，也更适合后续推荐排序与统计分析。这个思路和 PyABSA 所支持的任务路径并不冲突，只是你在研究设计上选择更可控的落地方式。([pyabsa.readthedocs.io][4])

### 7.1.6 知识库存储结构

建议至少建立以下 4 张核心表：

#### hotel 表

* hotel_id
* 酒店名称
* 城市
* 区域
* 星级
* 价格区间
* 品牌
* 酒店简介

#### review 表

* review_id
* hotel_id
* 评论时间
* 评分
* 原始评论文本
* 清洗后文本

#### aspect_sentiment 表

* hotel_id
* review_id
* sentence_id
* aspect_category
* sentiment_label
* opinion_span
* confidence

#### evidence_index / vector_store

* text_id
* hotel_id
* sentence_text
* aspect_category
* embedding
* metadata

### 7.1.7 向量检索方案

BGE v1.5 系列模型提供中文 embedding 能力，而其官方模型卡同时明确建议：embedding 模型先召回后，再使用 `bge-reranker-base` 或 `bge-reranker-large` 对 top-k 结果进行重排，因为 reranker 通常比纯 embedding 更强。([Hugging Face][5])

因此推荐配置为：

* 召回模型：`BAAI/bge-small-zh-v1.5` 或 `bge-base-zh-v1.5`
* 重排模型：`bge-reranker-base`

### 7.1.8 向量库选型

Faiss 官方文档将其定义为用于高效相似度搜索与聚类的 dense vector 检索库，支持大规模向量搜索与参数调优。对毕业设计这类中等规模实验而言，Faiss 很适合作为实验阶段的向量检索基础设施。([Faiss][6])

### 7.1.9 该模块的最终输出

这一模块最终应输出：

1. 酒店结构化信息表；
2. 评论与分句表；
3. 方面情感表；
4. 方面聚合统计表；
5. 向量索引；
6. 可用于 RAG 的证据片段集合。

## 7.2 模块二：用户偏好解析与澄清交互

### 7.2.1 模块目标

把自然语言需求转成可执行约束，并判断当前信息是否足够支持推荐。

### 7.2.2 推荐抽取的槽位

建议固定以下槽位：

* 城市
* 区域 / 商圈
* 预算上限 / 区间
* 入住时间
* 出行目的（出差 / 旅游 / 亲子 / 情侣等）
* 关注方面（安静、卫生、地铁近等）
* 硬约束
* 软偏好
* 不接受项

### 7.2.3 澄清交互的设计原则

当以下信息不足时，系统进入澄清模式：

* 城市缺失；
* 预算缺失；
* 区域与价格冲突明显；
* 用户表达过于模糊；
* 条件无法形成可执行候选集。

### 7.2.4 为什么这一块要微调

这部分最适合用 PEFT 做任务适配，因为它更像“行为模式学习”而不是“知识记忆”。PEFT 文档说明 LoRA 是通过在注意力等层中引入低秩适配矩阵来做参数高效微调；PEFT 的量化文档则说明结合 4-bit 量化与 LoRA，可以在更低资源条件下训练较大模型。([Hugging Face][7])

### 7.2.5 输出格式

建议这一步先输出 JSON，而不是直接输出自然语言，例如：

```json
{
  "city": "上海",
  "budget_max": 400,
  "trip_type": "出差",
  "focus_aspects": ["安静程度", "位置交通"],
  "hard_constraints": ["近地铁"],
  "clarification_needed": false
}
```

这样便于调试，也便于后续做偏好解析实验。

## 7.3 模块三：候选酒店筛选与排序

### 7.3.1 模块目标

不要让 LLM 直接从全量评论中“自由发挥”，而是先通过规则和统计方法缩小候选空间。

### 7.3.2 三级筛选流程

#### 第一步：结构化过滤

根据以下硬约束过滤：

* 城市
* 区域
* 预算
* 酒店类型

#### 第二步：方面加权排序

根据用户关注方面计算综合分，例如：

* 安静程度：0.35
* 卫生环境：0.25
* 服务质量：0.20
* 位置交通：0.20

#### 第三步：证据召回

对排序靠前的 Top-N 酒店，再从评论句向量库中召回最相关证据。

### 7.3.3 建议加入两个“很加分”的统计特征

#### 时间衰减

新评论权重更高，体现近期体验。

#### 争议度

若同一方面正负评论分歧很大，则视为“争议高”。

这会让系统更像一个真实推荐系统，而不是简单情感统计器。

---

## 7.4 模块四：RAG 证据检索与上下文组装

### 7.4.1 模块目标

从候选酒店中找出最能支持推荐理由的证据，而不是简单堆评论。

### 7.4.2 推荐的检索流程

1. 结构过滤出候选酒店；
2. 对候选酒店的评论句做 dense retrieval；
3. 用 BGE reranker 对 Top-K 证据重排；
4. 选取 3–8 条高质量证据进入上下文。

BGE 官方模型卡明确建议 embedding + reranker 的组合。([Hugging Face][5])

### 7.4.3 上下文组织方式

建议分为三部分：

#### A. 用户需求摘要

结构化约束转写成简洁说明。

#### B. 候选酒店摘要

每家酒店给出：

* 价格
* 区域
* 主要优点
* 主要风险

#### C. 证据片段

每条证据附带：

* 酒店名
* 方面类别
* 评论时间
* 评论句内容

---

## 7.5 模块五：LLM 推荐生成与多轮更新

### 7.5.1 模块目标

在给定候选酒店与证据的情况下，输出：

* Top-N 推荐；
* 对每家的推荐理由；
* 风险提示；
* 当证据不足时的谨慎说明。

### 7.5.2 模型选型建议

Qwen2.5 有 3B、7B 等多个指令模型版本；Qwen3-4B-Instruct-2507 官方模型卡显示其参数量为 4.0B，原生支持 262,144 上下文长度，并强调 instruction following、general capabilities 等能力。对你的任务而言，3B/4B 级模型更适合作为主实验模型，7B 可作为增强版对照。([Hugging Face][8])

### 7.5.3 为什么优先选择小模型

原因不是“小模型一定最好”，而是：

* 你的任务是受约束推荐，而不是开放世界复杂推理；
* 检索与候选排序对最终效果影响很大；
* 小模型更适合毕业设计快速迭代与多轮实验。

### 7.5.4 微调方式建议

采用 QLoRA：

* 4-bit 量化加载模型；
* 训练 LoRA 适配器；
* 优先做行为微调而非知识灌输。

Transformers 与 PEFT 文档都表明 4-bit/bitsandbytes 是当前较成熟的量化路径，而 PEFT 文档也指出量化 + PEFT 是单 GPU 训练大模型的典型策略。([Hugging Face][9])

### 7.5.5 建议微调的任务类型

你的 SFT/PEFT 数据不要只做“用户需求 -> 最终推荐文案”，而应该覆盖：

1. 偏好解析样本
2. 澄清提问样本
3. 证据式推荐样本
4. 约束遵守样本
5. 证据不足时的谨慎回复样本

### 7.5.6 输出要求

最终输出建议始终遵守：

* 不虚构酒店；
* 不虚构设施与地理细节；
* 推荐理由必须可被给定证据支持；
* 证据不足时显式说明不确定。

## 8. 整体编排框架建议

### 8.1 为什么不建议直接用“开放式 Agent”

因为你的系统流程高度可定义：

* 输入需求；
* 判断是否需要澄清；
* 解析槽位；
* 候选过滤；
* 证据检索；
* 推荐生成；
* 接收反馈；
* 重新执行。

这更接近一个**固定状态机**，而不是任意工具调用的 agent。

### 8.2 推荐编排方案

* **LangGraph**：主流程编排与状态管理
* **LangChain 组件**：Prompt、retriever、LLM 调用、memory 工具化

LangGraph 官方说明其适合构建有状态、可持久化、可调试的长生命周期工作流；这和你的多轮推荐任务是契合的。([LangChain Docs][1])

## 9. 两个创新点如何落到实验上

---

## 9.1 创新点一：细粒度方面知识库 vs 原始评论直接 RAG

### 研究问题

细粒度方面知识库是否比原始评论直接 RAG 更能提升推荐解释的可信度与可验证性？

### 对应实验比较

* **Raw Review RAG**：直接从原始评论句中召回
* **Aspect-KB RAG**：先做方面归一化与方面情感组织，再做召回与证据组合

### 主要观察指标

* Recall@K
* nDCG@K
* Evidence Relevance
* Citation Precision
* Citation Coverage
* Groundedness / Faithfulness
* Hallucination Rate
* 用户主观可信度

RAGAs 与 LangSmith 的评估框架都支持你把该问题拆成“检索质量 + groundedness + 最终输出质量”三个层面。([ACL Anthology][2])

### 预期结论方向

若 Aspect-KB RAG 在检索相关性、证据覆盖率和 groundedness 上明显更优，就可以支撑你的第一条创新点。

---

## 9.2 创新点二：PEFT 的收益主要体现在会话行为

### 研究问题

PEFT 微调对会话式推荐的主要收益，是否体现在偏好解析和澄清交互，而非事实性知识？

### 对应实验比较

* **Base-LLM**
* **PEFT-LLM**

### 建议拆成三个子任务

#### A. 偏好解析

输入用户话语，输出结构化槽位。

#### B. 澄清提问

判断该不该问、问什么。

#### C. 固定证据条件下的推荐生成

给 Base 和 PEFT 模型相同的候选酒店和相同证据，比较它们：

* 约束遵守能力；
* 表达组织能力；
* groundedness；
* 自然度。

### 预期结论方向

若 PEFT 在 Slot F1、澄清质量、约束遵守、自然度等方面显著提升，但在固定证据条件下对“事实正确率”的提升不明显，那么第二条创新点就能成立。

CRS 研究综述和关于 LLM 时代 CRS 评估的论文都指出，单纯比较“是否命中标准推荐项”是不够的，会低估系统在交互能力和解释性上的价值。([Springer][10])

## 18. 可直接作为开题/中期汇报的总结性表述

本文拟构建一个面向酒店评论场景的会话式推荐系统原型，整体采用“细粒度方面知识库 + RAG + PEFT 微调小模型”的技术路线。其中，知识库负责承载酒店事实与评论证据，RAG 负责根据用户需求完成候选证据检索，PEFT 微调的小语言模型负责偏好解析、澄清交互与证据式推荐生成。系统工程上采用有状态工作流组织推荐过程，避免使用过度开放的 agent 结构，以提升可控性与可解释性。实验上，本文采用 2×2 因子设计，系统比较“原始评论直接 RAG / 细粒度方面知识库 RAG”与“基础模型 / PEFT 微调模型”两类自变量对检索质量、解释 groundedness、偏好解析能力、澄清交互质量以及用户主观感知的影响，从而分别回答“知识组织形式是否提升推荐解释可信度”与“PEFT 的主要收益是否体现在会话行为而非事实知识”这两个核心研究问题。相关技术选型与评估思路可分别由 LangGraph/LangSmith、PEFT/量化文档、PyABSA、BGE 模型卡、Faiss 文档以及 RAGAs 与 CRS 评估研究提供方法支撑。([LangChain Docs][1])

### 阶段二：评论详情采集（解决“数据重叠与长尾分布”）

此阶段目标是解决“好评全是废话”和“数据重复”问题。针对每一家选中的酒店，执行以下 **瀑布流** 逻辑。

**目标配额：每家酒店 Max 300 条 (注意去除商家回复评论的内容,仅提取用户评论内容)。**

#### 步骤 1：负面警示池 (Negative Pool) —— **优先级最高**

* **筛选条件：** 标签选“差评(1-2分)” + “中评(3分)”。
* **排序：** 按时间（最新）。
* **采集量：** 上限 **100条**（不足 100 则全取）。
* **目的：** 挖掘“隔音差”、“卫生死角”等具体缺陷。
* **动作：** 存入数据库，记录 `Review_ID` 到去重集合。

#### 步骤 2：高质量证据池 (Evidence Pool) —— **优先级次之**

* **筛选条件：** 标签选“有图/视频”。
* **排序：** 按时间（最新）。
* **采集量：** 上限 **150条**。
* **去重逻辑：** 爬取时检测 `Review_ID`。如果已在“步骤1”中存在（即带图的差评），则**跳过**；否则存入。
* **目的：** 有图评论通常包含设施细节（床软硬、早餐种类），是 RAG 检索“环境描述”的最佳素材。

#### 步骤 3：时效性补全池 (Latest Pool) —— **优先级最低**

* **筛选条件：** 标签选“全部”。
* **排序：** 按时间（最新）。
* **采集量：** **填满剩余配额**（Target 300 - 已采数量）。
* **去重逻辑：** 检查 `Review_ID` 是否在“步骤1”或“步骤2”中存在？若存在则跳过。
* **目的：** 确保模型知道酒店“最近一周/一月”的状态。

#### 步骤 4：长尾熔断机制 (Circuit Breaker)

* **前置检查：** 进入页面前，读取总评论数。
* 若 `Total_Count < 50`：**直接放弃该酒店**（信息密度过低，易产生幻觉）。
* 若 `Total_Count` 在 50-300 之间：全量爬取。

| **score_clean** | `.starscore` -> 清洁程度 | 解析 `width: X%` 转换为 1-5 分。 |

| **score_value** | `.starscore` -> **性价比** | **关键字段**。替代具体价格，回答“值不值”。 |
| **tags** | `.review-tag` | 提取“位置好”、“服务热情”等标签。 |

## 建议 1：把数据明确拆成“三库”，不要混用

用途：

* 而是从 KB 中**自动构造 / 半自动构造任务样本**

建议样本类型：

* 偏好解析：自然语言 → JSON 约束

### C. 评测集（Eval）

## 建议 2：你当前的评论瀑布流策略“太偏负面+太偏带图”，适合做证据库，不适合直接做酒店画像

你当前每家酒店 300 条评论的大致分布是：

* 负面/中评：最多 100
* 有图/视频：最多 150
* 其余最新：补满

这个策略对“发现问题”很强，但会带来一个结构性偏差：

### 风险

系统会学到一个**过度风险敏感**的酒店画像：

* 很容易过度强调噪音、卫生死角、设施老旧
* 容易低估“普通住客大多数真实体验”
* 如果拿它直接做酒店层面的方面聚合，酒店分数会偏悲观
* 如果拿它直接做微调，模型回答会更容易“谨慎过头、挑刺过多”

### 更适合课题的改法

我建议你把每家酒店评论改成**四池采样**，而不是三池：

#### 池 1：负面警示池（20%–30%）

保留，用来挖缺点和风险点。

#### 池 2：证据强化池（20%–25%）

有图/视频、长评论、细节充分评论，用来做可解释证据。

#### 池 3：主流体验池（35%–45%）

从“全部评论”里按时间抽样，保留更接近真实分布的普通评论。

#### 池 4：方面补偿池（10%–15%）

针对该酒店评论中较少出现但任务很关键的方面补采，比如：

* 安静程度
* 亲子设施
* 接送服务
* Wifi/办公体验

### 为什么要这样改

因为你后面既要做：

* RAG 检索
* 方面聚合
* 酒店排序
* 微调样本构造

这几件事对评论分布的要求并不一样。
你现在的瀑布流更像“故障发现采样”，不太像“推荐系统知识采样”。

## 建议 3：酒店候选集不要只按“Top 排名”抓，要预留“随机样本”和“低评论样本”

你现在的酒店候选集构建已经有区域分层、价格分层，这很好。
但“每个价位抓 Top 评分 / Top 销量 / Top 综合”会继续带来两类偏差：

### 偏差 1：平台排序偏差

Top 结果本身受平台推荐逻辑影响，可能已经被广告、转化率、品牌效应强化过。

### 偏差 2：幸存者偏差

你抓到的会更多是“本来就做得不错、评论很多、运营成熟”的酒店。

### 更好的做法

建议在每个“商圈 × 价位档”里改成：

* **50% 头部样本**：保留你当前的 Top 逻辑
* **30% 随机样本**：在前若干页候选里随机取
* **20% 长尾样本**：选择评论少、排名靠后但仍有基础信息的酒店

### 为什么这很重要

因为你的系统不仅要会“推荐好酒店”，还要会：

* 识别证据不足
* 诚实说“不确定”
* 区分“真差”和“只是评论少”

这类能力在会话推荐里很重要，而 PEPPER 和相关 CRS 评估工作也正是在强调：系统不能只在“目标明确且头部候选充分”的情况下表现好，还要能处理偏好逐步暴露和信息不足的真实互动。([OpenReview][4])

### 对你当前“评论数 < 50 就放弃”的建议

这个规则不适合全部执行。
我建议：

* **作为知识库主集合**：可以放弃绝大多数 <50 评论酒店
* **但保留 10–15 家低评论酒店作为“低证据测试集”**

用途：

* 测系统会不会乱编
* 测“证据不足时能否保守回答”
* 测你的第二条创新点里模型的行为约束能力

# 三、按你现有三阶段策略，逐段给你优化建议

## 阶段一：酒店候选集构建

### 你当前方案的优点

* 区域分层很合理，覆盖了不同用户任务场景
* 价格分层也合理，适合推荐任务中的预算约束

### 我建议你补 3 个字段

除了你已有的 `Hotel_ID / Name / Address / Base_Price / Price_Range / Lat/Long`，再补：

* `crawl_date`：抓取日期
* `district / business_area`：标准化后的区域标签
* `hotel_rank_at_crawl`：抓取时在搜索结果中的排序位置

### 为什么要加

因为酒店价格和排序都强时变。
后面如果你做误差分析，会很需要区分：

* 这是酒店本身问题
* 还是抓取时点造成的价格/排序偏差

### 关于价格字段的一个关键提醒

`Base_Price` 不能被当成长期真值。
更适合的用法是：

* 保存抓取时价格快照
* 额外派生 `price_bucket`（如 0–300 / 300–600 / 600–1200 / 1200+）
* 推荐时优先按价格桶和价格快照日期解释

因为你后面真正要回答的不是“永远 438 元”，而是“这家酒店大致属于什么价位、是否符合预算”。

---

## 阶段二：评论详情采集

### 1）保留 `Review_ID` 去重是不够的

很多平台会出现：

* 轻微改写的重复评论
* 多房型模板评论
* 相同意思的短句重复

所以建议你在 `Review_ID` 去重之外，再做一级**近重复去重**：

* 文本标准化后 exact match 去重
* SimHash / MinHash 去重
* 或句向量相似度阈值去重（例如 >0.95）

### 2）不要只存 review 级文本，必须存 sentence 级证据

这一点对 RAG 很关键。

建议你在采集后立刻做：

* 评论分句
* 每句保留 `sentence_id`
* 每句保留来源 `review_id`
* 每句保留 `hotel_id`
* 后续 ABSA 在句子级运行

因为你后面真正拿来检索的最好不是整条评论，而是**句子级证据片段**。
RAGAs 的 faithfulness/groundedness 思路，本质上也要求“回答中的断言能被具体上下文支持”，句子级证据比整条评论更适合做这个验证。([ACL Anthology][5])

### 3）“有图/视频”不一定等于“高质量证据”

这类评论确实更容易包含设施和环境细节，但也常有两类噪声：

* 只晒图不讲体验
* 图文不一致、文本极短

所以我建议把“Evidence Pool”的筛选标准改成：

* 有图/视频
* 或文本长度超过一定阈值
* 或含方面关键词/标签
* 或有明确房型/设施描述

也就是把它改成“**可解释证据池**”，而不是“媒体评论池”。

## 阶段三：细粒度字段映射

### 你当前字段设计是对的，但还少了 5 个很关键的字段

建议补充：

* `hotel_id`
* `sentence_id`
* `aspect_category`（后处理得到）
* `crawl_time`
* `source_hash / page_hash`

### 为什么值得加

因为这会直接决定你后面能不能顺利做：

* aspect knowledge base
* evidence backtracking
* 误差定位
* 引用可核验性分析

### 对 `score_clean / score_service / score_value` 的建议

这类字段很有价值，但不要把它们视为“真实标签”，更适合当作：

* 弱监督信号
* 辅助排序特征
* 与评论文本交叉验证的补充信息

原因是平台星级或局部评分可能本身就受页面展示逻辑和样本量影响，并不一定能直接代表细粒度方面质量。

### 2. 每个区域只保留“价格分层”，不要再做太复杂的搜索矩阵

每个区域内，只保留 3 个价位层：

* **经济型：0–300**
* **舒适型：300–600**
* **中高端：600+**

每个“区域 × 价位层”抓 **10–15 家酒店** 即可。

### 推荐抽样方式

在每个“区域 × 价位层”里：

* **60% 抓头部酒店**（评分或销量靠前）
* **40% 抓非头部酒店**（随机或靠后一些）

这样做的目的不是“更均匀好看”，而是避免知识库只会“懂热门酒店”，不会处理普通酒店或证据不足酒店。

---

### 3. 评论不要每家抓 300 条，改成每家抓 80–150 条高质量评论

这是最值得精简的一步。

建议改为：

* **每家酒店目标评论数：80–150 条**
* **评论总量大致：1.2 万 – 2 万条**

这已经够做：

* 方面情感抽取
* 向量检索
* 证据片段构建
* 酒店画像聚合
* 微调样本生成

而且后续评论分句后，实际可用的 sentence-level 证据会远多于 review 数量。

### 为什么可以缩到这个规模

因为你真正需要的是**高质量、可检索、可验证的评论证据**，不是单纯“评论总数很大”。
QLoRA 论文明确提到，小而高质量的数据可以带来很强的效果；PEFT 文档也说明量化 + PEFT 的价值在于用较低成本完成任务适配。([arXiv][1])

---

### 4. 原始评论不要直接拿去微调，只拿来建知识库

这条必须保留。

你的数据用途应该改成：

* **原始评论 → 知识库 / 检索库**
* **知识库 → 再构造 SFT/PEFT 任务样本**

不要把评论直接当“用户问答训练对”。

因为你课题里模型微调要学的是：

* 偏好解析
* 澄清提问
* 约束遵守
* 基于证据的推荐解释

而不是背评论内容本身。
这也更符合我们前面定下来的路线：**知识外置，能力内化**。([arXiv][1])

你原来三池方案里，负面评论和有图评论权重偏高，适合发现问题，但不适合直接形成酒店整体画像。
时间有限时，我建议你只保留下面这个更简单的版本：

#### 池 3：高细节证据池

* 来源：有图/长评论/标签丰富评论

# 五、当前名单下，我建议你做“功能区再平衡”，但不靠补酒店，而靠“控制评论爬取深度”

这是最适合你当前阶段的补救方式。

因为从你给出的名单看，**交通枢纽区、老城文化区、会展活动区、CBD 商务区都比较丰富**，而度假亲子区也不少。
如果你对每家酒店都爬同样多评论，知识库会被“评论量本来就巨大的头部区”和“温泉/机场/南站”这些强场景淹没。

### 所以建议你对功能区设评论总预算

例如：

* CBD 商务区：25–30 家
* 老城文化区：25–30 家
* 交通枢纽区：20–25 家
* 会展活动区：20–25 家
* 度假亲子区：15–20 家

然后在区内再按评论深度分档。

这样做的好处是：

* 不用回头重爬酒店列表
* 能把当前名单变得更“适合论文实验”
* 避免知识库被温泉/机场类场景过度主导

# 六、评论爬完之后，你要立刻做的后处理流程

---

## 1）去重

至少做两层：

### 第一层

`review_id` 去重

### 第二层

文本近重复去重：

* 去空格/表情/模板短语
* 再做 exact match
* 还能做一次高相似度去重

这一步很重要，因为酒店评论里经常有：

* 模板化好评
* 轻微改写的重复评论
* 房型复制评论

---

## 2）分句

不要只留 review 级文本，必须生成 sentence 级证据表。

因为后面 RAG 更适合检索“句子级证据”，而不是整条评论。
RAGAs 的设计就是围绕 context relevance、faithfulness 等维度来评估 RAG，这种评估天然更适合句子级证据回溯。LangSmith 的 RAG 评估教程也把 retrieval quality 与 answer quality 拆开来测。([LangChain Docs][2])

### 建议生成字段

* `sentence_id`
* `hotel_id`
* `review_id`
* `sentence_text`
* `create_time`
* `room_type`
* `tags`

---

## 3）方面体系固定

这里不要贪多，直接定成 6–8 个方面：

* 位置交通
* 卫生环境
* 服务质量
* 设施舒适度
* 性价比
* 安静程度
* 亲子设施
* 商务便利性

后面 ABSA 或规则映射都围绕这套固定体系做，最稳。

## 第二步

### 建议优先构造 4 类样本

#### 1. 偏好解析样本

输入：
“下周去广州出差，预算 400 左右，离地铁近一点，晚上安静。”

输出：
结构化 JSON

#### 2. 澄清提问样本

输入：
“帮我推荐个广州住的地方，舒服点。”

输出：
模型应该追问预算、区域或出行目的。

#### 3. 证据式推荐样本

输入：
用户约束 + 候选酒店 + 评论证据

输出：
推荐结果 + 推荐理由 + 风险提示

#### 4. 反馈更新样本

输入：
“再便宜一点”“换更安静的区域”“不要公寓型”

输出：
更新后的推荐

QLoRA 论文明确指出，小而高质量的数据集可以得到很强的微调效果；这正说明你后面更应该做“任务对齐样本”，而不是继续堆原始评论。([arXiv][1])

# 九、基于当前名单的实验实施方案

## 实验前的数据切分

从现在开始，把酒店分成：

* **Train hotels**
* **Val hotels**
* **Test hotels**

建议比例：

* 70% 训练
* 10% 验证
* 20% 测试

### 切分原则

* 按 `hotel_id` 切
* 各功能区尽量都有 train/test 覆盖
* 特殊类型酒店优先留部分到 test

---

## 创新点 1 的实验

比较：

* Raw Review RAG
* Aspect-KB RAG

要求：

* 用同一批测试酒店
* 同一候选集
* 同一 LLM
* 同一 prompt
* 同一 top-k

重点看：

* 检索相关性
* 证据覆盖
* groundedness
* hallucination

LangSmith 的 RAG 评估教程就是按 retrieval quality、answer relevance、answer accuracy 来拆分评估流程的；RAGAs 也强调 reference-free 的 context precision、faithfulness 等指标。([LangChain Docs][2])

---

## 创新点 2 的实验

比较：

* Base LLM + fixed evidence
* PEFT LLM + fixed evidence

重点不是看“谁知道更多酒店事实”，而是看：

* 偏好解析准确率
* 澄清是否合理
* 约束是否遵守
* 解释是否更有条理
* 是否更少乱编
