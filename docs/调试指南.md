# 本地浏览器爬虫调试指南

本文用于本地调试和验证飞猪酒店评论爬虫，重点覆盖 Chrome 调试模式、飞猪登录、酒店列表/评论采集、验证码自动化、中断恢复与数据校验。

## 1. 环境准备

### 1.1 Python 环境检查

先进入项目根目录：

```bash
cd D:\VSCodeWorkspace\HotelReviewCrawling
```

检查 Python 版本，要求 `Python 3.10+`：

```bash
python --version
```

预期输出示例：

```text
Python 3.12.8
```

如果项目使用仓库内现成虚拟环境，可直接激活：

```bash
# Windows
venv312\Scripts\activate
```

如果你自己新建虚拟环境，可使用：

```bash
# Windows
python -m venv venv
venv\Scripts\activate

# macOS / Linux
python3 -m venv venv
source venv/bin/activate
```

### 1.2 依赖安装验证

安装依赖：

```bash
python -m pip install -r requirements.txt
```

检查关键依赖：

```bash
pip show DrissionPage SQLAlchemy pydantic-settings loguru tenacity
```

也可以直接运行单元测试做快速验证：

```bash
python -m pytest tests/ -v
```

预期结果：

- 能正常导入依赖
- `tests/test_anti_crawler_captcha.py` 与 `tests/test_captcha_recovery.py` 能执行
- 没有 `ModuleNotFoundError`

### 1.3 数据库连接测试

先复制环境变量模板：

```bash
# Windows PowerShell
Copy-Item .env.example .env

# macOS / Linux
cp .env.example .env
```

重点检查 `.env` 中以下配置：

| 配置项 | 说明 | 默认值 |
| --- | --- | --- |
| `DB_HOST` | PostgreSQL 主机 | `localhost` |
| `DB_PORT` | PostgreSQL 端口 | `5432` |
| `DB_NAME` | 数据库名 | `hotel_reviews` |
| `DB_USER` | 数据库用户 | `postgres` |
| `DB_PASSWORD` | 数据库密码 | `your_password` |
| `CHROME_DEBUG_PORT` | Chrome 调试端口 | `9222` |
| `CHROME_USER_DATA_DIR` | Chrome 用户目录 | `C:/selenium/automation_profile` |

快速测试 PostgreSQL：

```bash
psql -h localhost -p 5432 -U postgres -d hotel_reviews -c "SELECT 1;"
```

初始化数据库结构：

```bash
psql -h localhost -p 5432 -U postgres -d hotel_reviews -f database/init_db.sql
```

预期输出示例：

```text
 ?column?
----------
        1
(1 row)
```

### 1.4 Chrome 浏览器准备

要求：

- 使用本机已安装的 Chrome 浏览器
- 不要复用日常工作资料目录，建议单独的调试用户目录
- 首次调试前关闭所有已打开的 Chrome 调试实例

建议准备独立目录：

| 平台 | 推荐目录 |
| --- | --- |
| Windows | `C:\selenium\automation_profile` |
| macOS | `/tmp/chrome_debug` |
| Linux | `/tmp/chrome_debug` |

截图说明：

- 正确状态应看到一个全新的 Chrome 窗口
- 地址栏可正常访问网页

## 2. Chrome 调试模式启动

### 2.1 不同平台启动命令

```bash
# Windows
chrome.exe --remote-debugging-port=9222 --user-data-dir="C:\selenium\automation_profile"
```

```bash
# macOS
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222 --user-data-dir="/tmp/chrome_debug"
```

```bash
# Linux
google-chrome --remote-debugging-port=9222 --user-data-dir="/tmp/chrome_debug"
```

如果 Chrome 不在 PATH 中，Windows 可改为完整路径：

```bash
"C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222 --user-data-dir="C:\selenium\automation_profile"
```

### 2.2 调试端口验证

Chrome 启动后，访问：

```text
http://127.0.0.1:9222/json/version
```

或者命令行验证：

```bash
curl http://127.0.0.1:9222/json/version
```

预期输出示例：

```json
{
  "Browser": "Chrome/136.0.xxx.x",
  "webSocketDebuggerUrl": "ws://127.0.0.1:9222/devtools/browser/..."
}
```

### 2.3 用户数据目录说明

- `--user-data-dir` 会保存 Cookie、LocalStorage、会话状态
- 爬虫通过 `DrissionPage` 接管这个浏览器实例，而不是新开匿名窗口
- 只要同一个目录不被清空，飞猪登录状态通常可以复用
- 如果登录状态异常，先关闭 Chrome，再删除该目录重新登录

截图说明：

- `127.0.0.1:9222/json/version` 页面应返回 JSON，而不是连接失败

## 5. 分阶段调试

### 阶段一：酒店列表爬取测试

#### 5.1 单个商圈测试命令

先做最小范围验证：

```bash
python main.py --mode hotel_list --region "CBD商务区"
```

说明：

- `main.py` 会调用 `HotelListCrawler.crawl_region()`
- 每个商圈会走“分层抽样 + 弹性补位”
- 页面导航后会自动检查验证码

#### 5.2 预期输出说明

你应在控制台或日志中看到类似信息：

```text
开始爬取酒店列表
成功连接到Chrome浏览器: 127.0.0.1:9222
导航到: https://hotel.fliggy.com/hotel_list3.htm?...businessAreaId=...
功能区 CBD商务区 爬取完成，共 XX 家酒店
```

还可能看到：

- 地图容器加载日志
- 价格档过滤日志
- 页面翻页日志
- 已存在酒店去重日志

#### 5.3 常见错误排查

| 现象 | 常见原因 | 处理方式 |
| --- | --- | --- |
| 浏览器连接失败 | Chrome 未用调试模式启动 | 重新按第 2 节启动 Chrome |
| 未知功能区 | `--region` 名称不匹配 | 使用代码中的功能区名称，如 `CBD商务区` |
| 页面卡住无数据 | 飞猪页面未加载完成或页面结构变化 | 先手工刷新页面，再检查日志中的选择器报错 |
| 爬取数量异常偏少 | 触发过滤阈值或价格档不匹配 | 检查 `MIN_REVIEWS_THRESHOLD` 与日志过滤信息 |

#### 5.4 验证码触发场景

以下场景更容易触发验证码：

- 刚登录后立即高频翻页
- 多次重复切换筛选条件
- 短时间内重复运行相同商圈爬取
- 网络抖动导致飞猪验证页反复刷新

```sql
ORDER BY created_at DESC
LIMIT 20;
```

确认点：

- `region_type` 已写入
- `business_zone_code` 已写入
- `price_level` 已写入
- 没有明显重复酒店

截图说明：

- 应看到 `hotels` 表中新增了最近写入的酒店记录

### 阶段二：评论爬取测试

#### 5.6 单个酒店测试命令

选择一个已有酒店 ID，例如：

```bash
python main.py --mode reviews --hotel-id 10019773
```

#### 5.7 双池策略与质量控制验证

评论采集当前按双池执行：

| 顺序 | 池名 | 目标 |
| --- | --- | --- |
| 1 | `negative` | 先按动态负评软目标主采，再接受实际短缺 |
| 2 | `positive` | 正向评论人工辅助翻页提取，并做质量筛选 |

批量模式说明：
- `python main.py --mode reviews --all` 默认执行“负评自动 + 正评半自动”
- `python main.py --mode reviews --all --negative-only` 可降级为仅负评批量采集
- 人工接管前程序会先滚到页面底部，并按 `REVIEW_POSITIVE_MANUAL_PREWAIT_SECONDS` 预等待
- 提示出现后若长时间无输入，会按 `REVIEW_POSITIVE_MANUAL_REMINDER_SECONDS` 周期提醒

日志中应出现：

```text
开始瀑布流采集酒店 10019773，动态目标 total=..., negative=..., positive=...
步骤1: 爬取负面警示池
步骤2: 采集正向评论，剩余总配额 ...
人工正向采集质量过滤摘要: ...
```

同时应关注：
- `filtered_quality_total` 是否过高：过高说明长度/信息量阈值可能过严
- `short_comment_ratio` 是否过高：过高说明放宽比例可能过大
- `high_quality_ratio` 是否过低：过低说明知识库证据密度不足

#### 5.8 预期输出说明

典型输出：

```text
开始爬取评论
开始瀑布流采集酒店 10019773，动态目标 total=100, negative=25, positive=75, review_count=500
评论池完成: pool=negative, crawled=..., saved=...
评论池完成: pool=positive, crawled=..., saved=...
酒店 10019773 采集完成，共 ... 条评论，negative=.../..., positive=.../..., negative_ratio=..., high_quality_ratio=..., short_comment_ratio=...
```

#### 5.9 常见错误排查

| 现象 | 常见原因 | 处理方式 |
| --- | --- | --- |
| 直接跳过酒店 | 评论总数低于阈值 | 检查 `MIN_REVIEWS_THRESHOLD` |
| 评论翻页停住 | 正向人工页未完成翻页或页面结构变化 | 检查终端提示并确认浏览器已切到下一页 |
| 保存数少于采集数 | 去重或单条校验失败 | 检查日志和数据库已有数据 |
| 某个池中断 | 验证码、页面超时或过期 | 查看第 6 节验证码日志 |

#### 5.10 验证码触发场景

- 进入评论页时
- 切换差评/好评筛选时
- 翻页加载更多评论时

#### 5.11 数据库验证方法

```sql
SELECT review_id, hotel_id, source_pool, review_date, LEFT(content, 40) AS content_preview
FROM reviews
WHERE hotel_id = '10019773'
ORDER BY created_at DESC
LIMIT 20;
```

确认点：

- `source_pool` 包含 `negative`、`positive`
- 正向评论主要来自人工辅助采集
- 去重后 `review_id` 唯一

截图说明：

- 应看到同一酒店评论来自不同池，且时间、文本内容、回复字段合理

## 6. 验证码自动化调试

### 6.1 验证码触发测试

#### 如何主动触发验证码

可以使用以下方法提高触发概率：

1. 连续运行酒店列表爬取
2. 在评论页快速切换筛选并翻页
3. 清理 `CHROME_USER_DATA_DIR` 后重新登录，短时间密集访问

注意：不要为了触发验证码而开启高并发或暴力请求，本项目本身是单浏览器串行调试。

#### 观察自动化处理过程

当检测到验证码时，日志会先出现：

```text
检测到验证码: #nc_1_n1z
```

随后观察浏览器：

- 页面出现滑块组件
- 滑块自动被按下和拖动
- 验证成功后页面继续进入列表页或评论页

#### 验证人类化轨迹

自动滑块实现包含：

- 三段式 S 曲线
- 随机抖动
- 超冲后回拉
- 非固定停顿时间

浏览器里应该看到的现象：

- 拖动不是匀速直线
- 有轻微停顿
- 末尾可能略微超出后修正

#### 检查重试机制

默认重试参数：

| 配置 | 默认值 |
| --- | --- |
| `CAPTCHA_MAX_RETRIES` | `3` |
| `CAPTCHA_SOLVE_TIMEOUT_SECONDS` | `120` |
| `CAPTCHA_COOLDOWN_SECONDS` | `180` |
| `CAPTCHA_REFRESH_RETRY_LIMIT` | `2` |

如果自动滑块失败或验证页超时，应看到带重试计数的异常信息。

#### 验证冷却停止

当达到最大重试次数后，应停止当前任务，而不是无限重试。

酒店列表侧预期日志：

```text
酒店列表爬取因验证码冷却停止: ...
请等待至少 180s 后再重试当前任务
```

评论任务侧预期日志：

```text
任务停止: <task_id> - 验证码进入冷却期，180s 后可重试
```

### 6.2 日志分析

#### 验证码检测日志

关注关键词：

- `检测到验证码`
- `导航期间验证码处理失败`

#### 自动滑块日志

关注关键词：

- `自动滑块验证成功`
- `自动滑块失败`

#### 重试日志

关注关键词：

- `重试 1/3`
- `重试 2/3`
- `重试 3/3`

#### 冷却停止日志

关注关键词：

- `因验证码冷却停止`
- `进入冷却期`
- `请等待至少`

#### 恢复日志

恢复运行后，重点观察：

- 酒店列表是否从后续页面继续
- 评论采集是否从未完成池继续
- `pending` 任务是否重新被执行

Windows 过滤错误日志示例：

```bash
Select-String -Path .\logs\crawler_2026-03-10.log -Pattern "验证码|冷却|重试|刷新"
```

## 8. 数据验证

### 8.1 数据库查询命令

```sql
-- 查看酒店数量
SELECT COUNT(*) FROM hotels;

-- 查看评论数量
SELECT COUNT(*) FROM reviews;

-- 查看任务状态
SELECT hotel_id, name, review_count, updated_at
FROM hotels
ORDER BY updated_at DESC
LIMIT 10;

-- 查看爬取日志
SELECT hotel_id, source_pool, COUNT(*)
FROM reviews
GROUP BY hotel_id, source_pool
ORDER BY hotel_id, source_pool;
```

### 8.2 数据质量检查

#### 酒店数据完整性

```sql
SELECT hotel_id, name, region_type, business_zone_code, price_level
FROM hotels
WHERE hotel_id IS NULL
   OR name IS NULL
   OR region_type IS NULL
   OR business_zone_code IS NULL
   OR price_level IS NULL;
```

预期：返回 `0` 行。

#### 评论数据完整性

```sql
SELECT review_id, hotel_id, content, source_pool
FROM reviews
WHERE review_id IS NULL
   OR hotel_id IS NULL
   OR content IS NULL
   OR source_pool IS NULL;
```

预期：返回 `0` 行。

#### 去重验证

```sql
SELECT review_id, COUNT(*)
FROM reviews
GROUP BY review_id
HAVING COUNT(*) > 1;
```

预期：返回 `0` 行。

#### 分层抽样验证

```sql
SELECT region_type, business_zone_code, price_level, COUNT(*)
FROM hotels
GROUP BY region_type, business_zone_code, price_level
ORDER BY region_type, business_zone_code, price_level;
```

检查点：

- 每个功能区下应有多个商圈
- 每个商圈应尽量覆盖多个价格档
- 如果数量不均衡，需要结合日志判断是否发生弹性补位

## 9. 常见问题排查

### 问题 1：Chrome 连接失败

**症状描述**

- `浏览器连接失败`
- `连接Chrome浏览器失败`
- 访问 `127.0.0.1:9222/json/version` 失败

**原因分析**

- Chrome 未以调试模式启动
- 调试端口不是 `9222`
- 端口被其他程序占用
- `CHROME_DEBUG_PORT` 与实际端口不一致

**解决方案**

1. 关闭所有 Chrome 调试实例
2. 重新按第 2 节命令启动
3. 先访问 `http://127.0.0.1:9222/json/version`
4. 再运行 `python main.py --mode test`

### 问题 2：验证码无法自动处理

**症状描述**

- 验证码出现但滑块不动
- 多次重试后进入冷却
- 页面显示“验证已过期，请刷新”

**原因分析**

- 飞猪验证码组件样式变化
- 页面过期或 DOM 未稳定
- 网络抖动导致验证回包失败
- 页面未处于可交互前台状态

**解决方案**

1. 保持浏览器窗口可见，不要最小化
2. 手工刷新验证页后再重试
3. 检查 `logs/crawler_*.log` 中的验证码日志
4. 必要时重新登录飞猪并重启 Chrome 调试实例

### 问题 3：数据库连接失败

**症状描述**

- `数据库连接失败，请检查配置`
- `psql` 无法连接

**原因分析**

- PostgreSQL 未启动
- 用户名密码错误
- 数据库不存在
- 本机防火墙或端口配置错误

**解决方案**

1. 用 `psql` 先做最小连接测试
2. 核对 `.env` 中 `DB_*` 配置
3. 确认数据库 `hotel_reviews` 已创建
4. 重新执行 `database/init_db.sql`

### 问题 4：页面元素找不到

**症状描述**

- 翻页失效
- 评论筛选按钮点击失败
- 酒店/评论数量明显异常

**原因分析**

- 飞猪页面结构调整
- 页面未完全加载
- 实际进入了验证码页或异常页

**解决方案**

1. 先手工在浏览器里确认当前页面内容
2. 检查是否触发验证码
3. 在关键位置临时加断点或 DEBUG 日志
4. 对照 `crawler/hotel_list_crawler.py`、`crawler/review_crawler.py` 中的选择器排查

### 问题 5：中断后无法恢复

**症状描述**

- 重新运行后仍从头开始
- 重复写入或重复尝试相同任务

**原因分析**

- 已保存数据未成功提交
- 调试时改动了数据库或清空了表
- 评论/酒店恢复前置条件未满足

**解决方案**

1. 先用 SQL 确认中断前数据是否存在
2. 查看 `hotels` 和 `reviews` 中是否已经写入部分结果
3. 检查是否误删了 `hotels` / `reviews` 数据
4. 对评论采集重点检查 `source_pool` 分布与去重结果

```sql
SELECT COUNT(*) FROM pg_stat_activity WHERE datname = 'hotel_reviews';
```
